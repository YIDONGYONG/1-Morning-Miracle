1% Morning Miracle：MVP 企画・設計書
1. サービス概要
どんなサービスか：意志力ではなく「仕組み（システム）」によって朝の行動を変革し、目標達成を支援する朝専用の習慣化アプリ。
誰の課題をどう解決するか：
対象：朝の時間を活用したいが、スマホの誘惑や意志力の限界で挫折してしまう人。
解決策：毎朝「1%の極小ルーチン」を積み重ねることで悪習慣を断ち切り、一流の成功者のような生産性の高いライフスタイルを構築する。

2. このアイデアはどこから生まれたか
2-1. きっかけとなった体験・感情
背景：『Atomic Habits（アトミック・ハビッツ）』や『ミラクルモーニング』等の自己啓発書に感銘を受ける。
気づき：一流のCEOやアスリートは、一貫した「朝のルーチン（瞑想・運動・日記など）」をシステム化して1日の成功を確定させている。
感情：根性論ではなく、「1%の小さな変化の複利効果」で人生を劇的に改善できるシステムを作りたいという強い情熱から着想。
2-2. なぜそれが「気になった」のか
原体験：働きながら独学でプログラミングを学び、エンジニアへのキャリアチェンジに挑戦した際、仕事と学習の両立の過酷さを当事者として痛感。効率的な学習と習慣化の仕組みの必要性を誰よりも理解していたため。
価値観：「他人に貢献し助けることで、巡り巡って自分もより大きな成長というリターンを得られる（Giverの精神）」という信念があり、同じように限られた時間で未来を変えようと奮闘する人の力になりたいと考えたため。

3. 課題の整理（核心への深掘り）
3-1. 表に見えている困りごと
朝のルーチンを毎日コツコツ継続できない。
朝活を始めても、すぐに三日坊主で終わってしまう。
3-2. 本当に解決したい課題は何か
なぜ長続きしないのか？
スマホやSNSの普及による現代人の深刻な集中力低下（成人ADHD化）。この脳疲労状態で、最初から「1時間勉強」などモチベーション最高潮時の高すぎる目標を立てるため。
なぜ高い目標だとダメなのか？
体調や気分が悪い日に未達成になると、完璧主義が邪魔をして「ゼロかヒャクか」の思考に陥り、全ての習慣を放棄するため。
なぜ放棄してしまうのか？
未達成への「挫折感」や「自己嫌悪」がトリガーとなり、行動を起こすこと自体が苦痛（ストレス）に変わるため。
➔ 本当に解決したい課題： 「完璧にできない自分」への挫折感から行動を完全停止してしまう心理的障壁と、スマホ依存による集中力低下を、意志力ではなく「システム」で克服すること。

4. 想定ユーザーについて
4-1. ペルソナ設定
年齢：30代（特に30代後半のキャリア転換期・リスタート層）
理由：20代より可処分時間が激減し、朝活への切実なニーズがある。挫折を経験したからこそ根性論ではない「仕組み」を求める世代だから。
生活状況：日中はハードな労働や現職をこなしつつ、早朝に「エンジニア転職」や「起業」を目指し猛勉強している限界状態の学習者。
理由：疲労度が高く、朝一番を自動化しないと、貴重な自己投資時間を二度寝やスマホに奪われる死活問題を抱えているため。
環境：起床直後、ベッドの中で最も誘惑に負けやすい「スマホの画面と対峙している瞬間」。
理由：エネルギーゼロの瞬間に、迷う余地のない「1分だけの超イージーな行動」へ身体を動かす必要があるため。
4-2. 自分とユーザーの距離
距離感：自分自身 / 過去の自分
理由：仕事と並行してプログラミングを独学し、必死に生活を変えようと努力してきた自身の経験・痛みと完全に一致するため。
4-3. 需要・存在確認
初期ユーザー：朝の時間を変えて人生を本気で変えたい社会人・学習者。
思い出す瞬間：目が覚めた瞬間、または前夜に「明日の朝こそ時間を無駄にしない」と決意する時。
既存の代替品：一般的なTo-Doリストアプリ（Todoistなど）、スマホの標準アラーム。
置き換える理由：既存アプリは「タスク消化」の義務感と意志力を求めるが、本アプリは「1%の極小実践」に特化し、確実に変化を実感できるため。
4-4. 導入・継続の流れ
導入：意志力の限界を悟り、「仕組み（システム）」による自己変革を求めて登録。
継続の仕組み：目標未達成の週は、自動的に目標難易度を下げる「ダウングレード回顧」を作動。完璧主義の罠（途切れたら諦める）を防ぎ、自己効力感を維持させる。

5. 既存サービス・競合調査
5-1. 似たサービスの調査
サービス名：Todoist（トドイスト）
概要：GTD理論に基づくタスク管理の王道アプリ。タスク整理や連続達成日数（ストリーク）の可視化に優れる。
5-2. それでも自分が作りたい理由（既存の不満・ズレ）
Todoistは未消化タスクが翌朝に「期限切れ（赤文字）」として蓄積される。これが朝一番のユーザーに「負債感」と「認知疲労」を与え、モチベーションを挫く。また、重い業務と「1分瞑想」が同列に並ぶため、習慣が「義務（仕事）」に感じられてしまう。
5-3. 差別化を一文で決める
タスク管理アプリの「期限切れのプレッシャー」を徹底排除し、エネルギーゼロでも動ける「極小の行動（1%）」のみに特化。失敗をハードル低下へと昇華させる挫折回避型アプローチ。

6. このサービスで提供したい価値
6-1. ユーザーの変化
行動：起床直後のダラダラスマホが消え、「1分瞑想」「5分散歩」等の善い行動が自然と定着する。
気持ち：小さな成功体験の連鎖により自己嫌悪が消え、爽快な自己効力感で1日をスタートできる。
考え方：目標達成に必要なのは根性ではなく、システムの構築と「1%の複利効果」であると確信する。
6-2. 価値を一文で表す
「朝の挫折を繰り返す人が、意志力に頼らない極小のシステムを通じて、一流の成功者のようなライフスタイルを確実に手に入れられるサービス」

7. このアプリで実現すること
MVPで作る機能
ユーザー認証・マイページ（人生の目標・ビジョン登録）
マイクロルーチン・カスタム（1分瞑想、5分散歩など極小行動の登録）
ワンタッチ完了チェック（ボタン1つで記録できる極シンプルUI）
1%成長ダッシュボード（連続達成日数（Streak）の可視化グラフ）
本リリースで作る機能
スマホウィジェット（ホーム画面に目標や「過去の悔しかった瞬間」を常時表示し、スマホ依存を抑止）
AIパーソナライズ提案（バイリズムに合わせ、ルーチン負荷を自動調整）
非同期型サイレント・コミュニティ（過度な交流ストレスなく、他人の継続数（Streak）のみで刺激を貰う）

8. このアプリの懸念点とその対策
懸念点①：朝の寝ぼけた状態で、アプリを開く（初動）こと自体が面倒になり離脱する
原因：どれほど優れた機能も、アプリが起動されなければシステムとして機能しないため。
対策：MVP段階では、アラーム音連動のWeb通知（Push API）やPWA化による常駐を検討。スマホのドックなど「一番押しやすい位置」への配置を促すUX設計にする。

9. 今後の展開・発展の方向性
機能追加：21日間連続達成で「1分瞑想 ➔ 3分瞑想」へ自動誘導する「漸進的負荷アップ機能」。
UI・体験：React＋Tailwindによる滑らかなアニメーションで、完了時のドーパミン（ご褒美感）を最大化。
データ活用：蓄積データから「ユーザーが最も挫折しやすい曜日（例：木曜朝）」を割り出し、前夜に先回りケア通知を送る。

10. 技術スタック（手段としての技術）
10-1. 使用予定の技術
フレームワーク：React / Ruby on Rails 7 (APIモード)
データベース：PostgreSQL
インフラ・環境：Docker（開発環境）
10-2. 技術選定の理由
Rails7（APIモード）で堅牢なデータ設計（Streak管理など）と爆速開発を両立。
フロントをReactでSPA化し、朝の短い時間にストレスを与えない「超高速な操作性」を実現。
チャレンジ・不安点：RailsとReactの完全分離アーキテクチャの構築、およびJWTトークン認証の実装コスト。
10-3. キャッチアップ不足の懸念と対策
懸念： 初めてのアプリ開発で、あれもこれもと機能を詰め込みすぎてハードルが高くなってしまうこと。
対策： まずはシンプルな機能から作り始めること。そして、「本当にユーザーが使う機能」だけを見極め、使われない無駄な機能は作らないこと。
